================================================================================
  1. PROJECT OVERVIEW
================================================================================

This is a full-stack Hostel Management System built using the MERN stack. It
allows two types of users — Students and Admins — to manage all aspects of
hostel life digitally. The system is organized around "hostel blocks," where
each admin manages students belonging to their specific block only.

The architecture follows a client-server model:
  - Frontend (Client): React.js SPA served via Vite dev server (port 5173)
  - Backend (Server):  Express.js REST API (port 5000)
  - Database:          MongoDB Atlas (cloud-hosted NoSQL database)
  - Cache/Session:     Redis (for refresh tokens and OTP storage)

Two user roles exist in the system:
  1. STUDENT — Registers, gets approved by admin, then accesses hostel services
  2. ADMIN   — One admin per hostel block, manages all students in that block

  3. FEATURES — STUDENT USER
================================================================================

3.1  REGISTRATION & LOGIN
     - Register with: Full Name, Student ID, Branch, College Email, Hostel
       Block, Room Number, Password, Parent Name, Parent Phone, Blood Group
     - Registration goes to "pending" state — must be approved by block admin
     - Login via Student ID + Password (only after admin approval)
     - Login via Google OAuth 2.0 (using college email, must be pre-approved)
     - Check Registration Status page (public — no login needed)
     - Forgot Password with email OTP verification (OTP sent via Nodemailer,
       stored in Redis with 10-min expiry)

3.2  STUDENT DASHBOARD
     - Overview cards showing: Total Attendance, Active Complaints, Mess 
       Leave Days
     - Attendance Trend chart (last 7 days bar chart via Recharts)
     - Complaints by Category (pie chart via Recharts)
     - Recent Announcements list (latest 5 from their hostel block)
     - Recent Approved Mess Leaves list

3.3  PROFILE MANAGEMENT
     - View full profile (name, student ID, branch, email, hostel block,
       room number, parent info, blood group)
     - Edit profile (update name, branch, parent name, parent phone)
     - Change Password (requires current password verification)

3.4  COMPLAINT SYSTEM
     - File new complaints with category selection:
       Electricity, Water, Mess, Fans, Lightbulb, Other
     - Add detailed description for each complaint
     - View all personal complaints with status tracking:
       Pending → Accepted → Resolved
     - Receive in-app notifications when complaint is resolved
     - Receive email notifications on complaint status changes

3.5  GUEST ROOM BOOKING
     - Browse 100 guest hostel rooms with real-time availability
     - Select date range (from - to) to check availability
     - Room status visualization: Available / Booked / Pending
     - Book a room by providing: Visitor Name, Relation, Room Number,
       Date Range, Purpose
     - Conflict detection prevents double-booking approved rooms
     - View all personal bookings (My Bookings page)
     - Receive notification when booking is approved

3.6  MESS LEAVE MANAGEMENT
     - Apply for mess leave with: Start Date, End Date, Reason
     - Date validation (end date must be after start date)
     - View all personal mess leave requests with status tracking
     - Statuses: Pending → Approved / Rejected
     - Receive in-app notification + email when leave is approved

3.7  INVOICES & FEES
     - View all personal invoices (individual + block-wide broadcast invoices)
     - See invoice details: Title, Description, Amount (₹), Due Date, Status
     - Track payment status: Pending / Paid
     - Receive notification when new invoice is generated

3.8  ANNOUNCEMENTS
     - View all announcements posted by hostel block admin
     - Announcements sorted by newest first
     - Also shown on dashboard (latest 5)

3.9  MESS MENU
     - View weekly mess menu for their hostel block
     - Organized by day (Monday–Sunday)
     - Shows: Breakfast, Lunch, Snacks, Dinner for each day

3.10 ATTENDANCE
     - View personal attendance history (last 30 records)
     - Track present/absent records by date

3.11 ROOM CHANGE REQUEST
     - Submit a room change request with: Desired Room Number, Reason
     - System checks if requested room is available (not occupied)
     - Only one pending request allowed at a time
     - View all personal room change requests and their status
     - Receive notification when approved (room auto-updated) or rejected
       (with admin remarks)

3.12 NOTIFICATIONS
     - In-app notification bell with unread count badge
     - Notifications for: Complaints resolved, Bookings approved, Invoices
       generated, Mess leaves approved, Room changes approved/rejected
     - Mark all as read / Mark individual as read
     - Latest 50 notifications displayed

================================================================================

4.1  ADMIN LOGIN
     - Login via Admin ID + Password (pre-seeded admin accounts)
     - Each admin is assigned to one hostel block (1 admin per block)
     - All admin operations are scoped to their hostel block only

4.2  ADMIN DASHBOARD
     - Overview stat cards showing:

       • Pending Guest Room Bookings
       • Open Complaints (pending + accepted)
       • Pending Mess Leave Requests

     - All data is block-scoped via MongoDB aggregation pipelines

     - View individual student full details (profile + all their complaints,
       attendance, mess leaves, invoices, guest room bookings)
     - Approve pending registration requests → Creates User account + sends
       approval email to student's college email
     - Reject registration requests → With reason + sends rejection email
     - Soft-delete (deactivate) students (sets isActive=false, isApproved=false)
     - Reactivate previously deactivated students

4.4  MANAGE COMPLAINTS
     - View all complaints from students in their block
     - Complaints show: Student Name, Room Number, Category, Description,
       Status, Date
     - Accept complaint and set resolution date/time
     - Mark complaint as resolved (with optional resolution note)
     - Update complaint status (pending/accepted/resolved)
     - On resolution: auto-sends in-app notification to student

4.5  MANAGE GUEST ROOM BOOKINGS
     - View all booking requests from students in their block
     - See booking details: Student info, Visitor Name, Relation, Room,
       Date Range, Purpose, Status
     - Approve or Reject bookings
     - Block-level authorization check (admin can only manage their block's
       students' bookings)
     - On approval: auto-sends in-app notification to student

4.6  MANAGE MESS LEAVES
     - View all mess leave requests from students in their block
     - See details: Student info, Start/End Date, Reason, Status
     - Approve or Reject leave requests
     - On approval: auto-sends in-app notification + email to student

4.7  MANAGE INVOICES
     - Create TARGETED invoice for a specific student
       (by Student ID, with title, description, amount, due date)
     - Create BROADCAST invoice for ALL students in the block at once
       (generates individual invoice copy for each student)
     - View all invoices in the block with student details
     - Mark invoices as PAID
     - On creation: auto-sends in-app notification to student(s)

4.8  MANAGE ANNOUNCEMENTS
     - Create new announcements with title and message
     - View all announcements for their block (sorted newest first)
     - Announcements are automatically visible to all students in the block

4.9  MANAGE MESS MENU
     - Set/Update weekly mess menu for their block
     - For each day (Monday–Sunday): Breakfast, Lunch, Snacks, Dinner
     - Upsert logic: creates new entry or updates existing for that day
     - Delete individual menu entries
     - One entry per day per block enforced by database index

4.10 MANAGE ATTENDANCE
     - Record attendance for any student in their block
       (by Student ID, Date, Status: present/absent)
     - View all attendance records for the block
     - View attendance for a specific date
     - View attendance statistics per student:
       Total records, Present count, Absent count, Attendance percentage

4.11 MANAGE ROOM CHANGES
     - View all room change requests from students in their block
     - See details: Student info, Current Room, Requested Room, Reason
     - Approve room change → Automatically updates student's room number
       in database + sends notification
     - Reject room change → With admin remarks + sends notification

4.12 ADMIN PROFILE MANAGEMENT
     - View admin profile (name, adminID, email, hostel block)
     - Update profile (name, email)
     - Change password (requires current password verification)

  5. AUTHENTICATION & SECURITY SYSTEM
================================================================================

5.1  DUAL-TOKEN JWT AUTHENTICATION
     - Access Token (15 min expiry):
       • Stored in-memory on frontend (not localStorage — XSS safe)
       • Sent via Authorization: Bearer header on every API request
       • Contains: userId, role (student/admin)
       • Verified by protectRoute middleware on protected API routes
     - Refresh Token (7 day expiry):
       • Stored in HTTP-only cookie (not accessible to JavaScript — XSS safe)
       • Also stored in Redis with same TTL for server-side validation
       • Used to silently get new access tokens when they expire
       • Auto-refresh handled by Axios response interceptor on 401

5.2  GOOGLE OAUTH 2.0
     - Students can login using Google with their college email
     - Passport.js GoogleStrategy looks up existing User by email
     - Only approved users can login via Google (isApproved check)
     - Returns JWT tokens same as normal login flow
     - OAuth success redirect handled by OAuthSuccess.jsx page

5.3  ROLE-BASED ACCESS CONTROL (RBAC)
     - ProtectedRoute component on frontend: checks authentication + role
       before rendering any route (student vs admin)
     - authorize middleware on backend: checks req.role from JWT
     - Every controller function validates role at the beginning
     - Block-level scoping: admins can only access data from their own
       hostel block (checked via hostelBlock field matching)

5.4  RATE LIMITING (3 Tiers)
     - Global: 200 requests / 15 min / IP (all API routes)
     - Auth: 15 requests / 15 min / IP (login, register routes)  
     - Sensitive: 5 requests / 1 hour / IP (forgot password, OTP)

5.5  PASSWORD SECURITY
     - All passwords hashed with bcryptjs (10 salt rounds)
     - Password comparison done server-side only
     - Forgot password flow:
       1. Student submits college email
       2. Server generates 6-digit OTP → stores in Redis (10 min TTL)
       3. OTP sent to email via Nodemailer
       4. Student submits OTP + new password
       5. Server verifies OTP from Redis → updates password → deletes OTP

5.6  CORS POLICY
     - Only configured frontend URL allowed (FRONTEND_URL from .env)
     - Credentials enabled for cookie-based refresh token flow
     - All other origins rejected with error


================================================================================
  6. NOTIFICATION SYSTEM
================================================================================

Two notification channels work in parallel:

6.1  IN-APP NOTIFICATIONS (MongoDB-backed)
     - Created via notify() utility in server/lib/notify.js
     - Stored in Notification collection with: student ref, type, title,
       message, isRead flag, optional refId for deep-linking
     - Types: complaint, mess_leave, booking, invoice
     - Frontend fetches via GET /api/notifications with unread count
     - Mark individual or all notifications as read
     - Notifications are silent (no push) — polled by frontend

6.2  EMAIL NOTIFICATIONS (Nodemailer via Gmail SMTP)
     - Registration approved/rejected → Email to student
     - Mess leave approved → Email to student  
     - Complaint resolved → Email to student
     - Password reset OTP → Email to student
     - Sent as HTML emails with styled content

================================================================================
  10. HOW THE SYSTEM WORKS — END-TO-END FLOW
================================================================================

STUDENT REGISTRATION FLOW:
  1. Student fills registration form on /register page
  2. Frontend sends POST /api/auth/login/student (or /api/registration)
  3. Server validates fields, checks for duplicates, hashes password
  4. Creates StudentRegistrationRequest document (status: "pending")
  5. Admin sees request on ManageStudents page (GET /api/admin/registrations)
  6. Admin clicks Approve → Server creates User doc, marks request "approved"
  7. Approval email sent to student's college email via Nodemailer
  8. Student can now login with Student ID + Password

AUTHENTICATION FLOW:
  1. Student/Admin submits credentials on /login page
  2. Server validates credentials, generates access + refresh tokens
  3. Access token returned in response body → stored in React state (memory)
  4. Refresh token set as HTTP-only cookie + stored in Redis
  5. Axios interceptor attaches Bearer token to every subsequent request
  6. When access token expires (15 min), 401 response triggers auto-refresh
  7. Axios interceptor calls POST /api/auth/refresh → gets new access token
  8. Original failed request is retried with new token — seamless to user

COMPLAINT RESOLUTION FLOW:
  1. Student creates complaint (POST /api/complaints)
  2. Admin views block complaints (GET /api/complaints/admin)
  3. Admin accepts complaint, sets resolution date/time
  4. Admin resolves complaint (PATCH /api/complaints/:id/resolve)
  5. Server creates in-app Notification for student
  6. Server sends resolution email to student's college email
  7. Student sees notification badge update, views resolved complaint

INVOICE BROADCAST FLOW:
  1. Admin creates broadcast invoice (POST /api/invoices, isBroadcast: true)
  2. Server finds ALL approved students in admin's block
  3. Creates individual Invoice document for each student
  4. Creates in-app Notification for each student
  5. Each student sees the invoice on their MyInvoices page
  6. Admin marks individual invoices as "paid" as students pay
