REDIS USAGE IN THIS PROJECT (WITH LINE NUMBERS)

1) WHY REDIS IS USED HERE
- Fast in-memory storage for short-lived and security-sensitive data.
- Used to keep server-side control over refresh tokens (session-like behavior for JWT auth).
- Used to store OTPs for password reset with automatic expiry.
- Helps invalidate tokens/OTPs quickly without waiting for JWT expiry.

2) WHAT REDIS IS DOING
A) Authentication token control
- Stores refresh tokens after login and OAuth login.
- Reads refresh token from Redis during /refresh to validate the cookie token.
- Deletes refresh token on logout.

B) Password recovery
- Stores OTP with 10-minute TTL.
- Reads OTP during reset-password verification.
- Deletes OTP after successful password reset.

C) Connection resilience
- Retries Redis connection with backoff.
- Stops retry after 5 attempts and allows app to continue (with Redis-dependent features unavailable).
- Tracks connection availability via a boolean flag.

4) KEY TAKEAWAY
Redis is the short-term security state layer in this app:
- refresh tokens in Redis make token revocation and server-side verification possible,
- OTPs in Redis provide quick, expiring password-reset verification,
- connection strategy in redis.js prevents full app crash when Redis is down.
